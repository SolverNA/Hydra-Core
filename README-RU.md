# Техническое задание: Проект "Hydra-Core"
**Версия:** 1.0 (Draft)  
**Статус:** Концептуальное проектирование  
**Стек:** Rust (Core), Kotlin/Java (Android/Desktop), Swift (iOS/macOS), JSON (Configs)  
**Лицензия:** MIT

---

## 1. Общее описание идеи
**Hydra-Core** — это кроссплатформенная библиотека и набор приложений, предназначенных для обхода систем DPI (Deep Packet Inspection) и ТСПУ. 

**Ключевая инновация:** В отличие от пассивных средств обхода, Hydra-Core использует стратегию **активного истощения ресурсов (Resource Exhaustion)**. Приложение имитирует поведение высоконагруженного домашнего маршрутизатора, генерируя смесь легитимного трафика пользователя и «мусорных» сессий, которые заставляют DPI-узлы выполнять дорогостоящие вычислительные операции, приводя к их троттлингу или отказу.

---

## 2. Архитектура системы

### 2.1. Уровни абстракции
Проект разделен на три слоя:
1.  **L3/L4 Core (Rust):** Низкоуровневая обработка пакетов, логика дефрагментации, генератор шума.
2.  **Orchestrator (Rust + FFI):** Логика управления конфигами (JSON), мониторинг качества связи (Health-checks), выбор стратегии.
3.  **Platform Wrapper (Java/Kotlin/Swift):** VPN-интерфейс ОС, UI, жизненный цикл приложения.

### 2.2. Сетевое взаимодействие
Приложение создает виртуальный сетевой интерфейс (**TUN/TAP**). Весь трафик устройства направляется в Rust-ядро.
* **Inbound:** Пакеты от приложений ОС.
* **Processing:** Ядро анализирует пакет (L3/L4), применяет модификации (Split, Desync, Fake DNS) и добавляет «паразитный» трафик.
* **Outbound:** Модифицированный поток уходит через стандартные сокеты ОС.

## 3. Техническая спецификация Core (Rust)

### 3.1. Модуль манипуляции пакетами (DPI-Evasion)
Реализация логики, аналогичной `zapret` и `GoodbyeDPI`, но на безопасном Rust:
* **TCP Segment Splitting:** Дробление SNI-запроса на части (1 байт + остаток), чтобы DPI не смог собрать заголовок.
* **Window Size Manipulation:** Установка крайне малого размера окна для замедления разбора пакетов цензором.
* **HTTP/TLS Desync:** Отправка фейковых пакетов с неправильной контрольной суммой или истекшим TTL, которые игнорируются целевым сервером, но обрабатываются ТСПУ.

### 3.2. Stress Engine (Генератор нагрузки)
Самая важная часть. Чтобы минимизировать нагрузку на канал пользователя ($bandwidth$), мы будем использовать **CPU-intensive** атаки на ТСПУ:
* **TLS Handshake Flood:** Генерация тысяч коротких сессий с валидными TLS-отпечатками (Firefox/Chrome), которые обрываются сразу после Client Hello. DPI обязан хранить состояние каждой сессии в памяти.
* **QUIC Churn:** Генерация UDP-пакетов, имитирующих протокол QUIC. ТСПУ сложно анализировать зашифрованный QUIC, и массовое появление таких сессий вызывает перегрузку алгоритмов классификации трафика.
* **Entropy Attack:** Отправка пакетов с высокой энтропией (выглядят как шифрованный трафик), заставляя DPI применять эвристики для определения протокола.

### 3.3. Модель "Виртуальный Маршрутизатор"
Для маскировки Rust-ядро должно:
* Рандомизировать поле **TTL** (Time To Live) для разных потоков (имитация разных устройств за NAT).
* Использовать разные **TCP Fingerprints** (разные размеры заголовков, опции SACK, Window Scale) для параллельных соединений.
* Смешивать трафик: 20% — реальные данные пользователя, 80% — «умный шум».

## 4. Система автоматизации и JSON-конфиги

### 4.1. Структура стратегии
Конфиг должен позволять обновлять логику без пересборки приложения.
```json
{
  "provider_id": "AS12345",
  "strategies": [
    {
      "name": "aggressive_yt_fix",
      "target_hosts": ["youtube.com", "googlevideo.com"],
      "evasion": {
        "split_pos": 2,
        "method": "fake_packet",
        "payload_type": "tls_v1.3"
      },
      "stress": {
        "enabled": true,
        "threads": 4,
        "mode": "handshake_flood"
      }
    }
  ]
}
```

### 4.2. Strategy Hunter (Охотник)
Алгоритм поиска:
1.  Берет базовый конфиг из JSON.
2.  Пингует «маяки» (Telegram, Google, YouTube).
3.  Если задержка > 500мс или пакеты теряются, начинает перебор параметров (сдвиг split, смена флагов).
4.  При нахождении рабочей комбинации — сохраняет её локально и отправляет (анонимно) на сервер агрегации (в будущем).

## 5. Платформенная реализация

### 5.1. Android (Java/Kotlin)
* Использование `VpnService` API.
* Передача дескриптора файла (FD) TUN-интерфейса в Rust-ядро через JNI.
* Foreground Service для предотвращения убийства процесса системой.

### 5.2. iOS (Swift + Rust)
* Использование `NetworkExtension` (Packet Tunnel Provider).
* Компиляция Rust-ядра как статической библиотеки `.a` с C-интерфейсом.
* **Важно:** Для iOS "Stress Engine" должен быть ограничен по потреблению энергии, чтобы система не закрыла расширение из-за "High Battery Usage".

### 5.3. Desktop (Windows/Linux)
* Использование драйвера `Wintun` (Windows) для создания интерфейса.
* Java-оболочка на **JavaFX** для визуализации графиков нагрузки и эффективности обхода.

## 6. Роадмап разработки (Milestones)

### Этап 1: "Ignition" (0-2 месяца)
* Разработка Core на Rust: реализация TUN-интерфейса.
* Портирование базовых методов `zapret` (split, desync) на Rust.
* Создание базового CLI-клиента для Linux.

### Этап 2: "Infection" (2-4 месяца)
* Реализация **Stress Engine**: генератор фейковых TLS-сессий.
* Написание JNI-прослойки для Android.
* Первая рабочая версия APK с кнопкой "Включить обход".

### Этап 3: "Hydra" (4-6 месяцев)
* Реализация системы автоматического подбора (Strategy Hunter).
* Внедрение JSON-конфигов и системы их удаленного обновления.
* Разработка iOS-версии (NetworkExtension).

### Этап 4: "Global Impact" (6+ месяцев)
* Публикация Core-библиотеки как MIT-крейта для Rust.
* Документация для сторонних разработчиков VPN.
* Масштабирование сети пользователей для достижения критической массы в 100к+ узлов.

## 7. Философия и юридический отказ
Проект Hydra-Core создается как исследовательский инструмент для изучения устойчивости сетей к цензуре. Разработчики не несут ответственности за использование инструмента в целях, нарушающих локальное законодательство. Однако мы верим, что **свобода информации — это фундаментальное право человека**, и техническое противодействие цензуре является формой цифровой самообороны.
